# 求职匹配浏览器插件 PRD

## 1. 背景

用户在 Boss 直聘、猎聘、拉勾等招聘网站浏览职位时，通常需要反复判断：

- 这个岗位和我的简历是否匹配。
- 我的简历里哪些经历适合突出。
- 岗位要求里有哪些风险点或缺口。
- 是否值得进一步沟通或投递。

这些判断依赖用户自己阅读职位描述、回忆简历内容、总结匹配点。插件的价值在于把这一步变成页面内的即时辅助分析，而不是替用户完成所有求职动作。

## 2. 产品定位

这是一个 Chrome 浏览器插件，在用户访问招聘网站职位详情页时自动触发侧边栏或浮层，基于用户上传的简历和求职意向，对当前职位进行匹配分析、风险提示和沟通建议。

第一版定位为“求职决策辅助工具”，不是自动投递工具，也不是自动聊天工具。

## 3. 目标用户

- 正在主动找工作的求职者。
- 需要快速筛选大量岗位的人。
- 对简历和岗位匹配度判断不够有信心的人。
- 希望根据不同岗位调整求职表达的人。

## 4. 核心问题

### 4.1 用户痛点

- 职位页面信息多，判断成本高。
- 同一份简历面对不同岗位时，用户不知道该突出哪些经历。
- 很多岗位看起来相似，但真实匹配点和风险点不同。
- 用户容易漏看硬性要求，例如学历、年限、城市、技能栈、行业经验。

### 4.2 插件要解决的问题

- 自动识别当前职位页面的关键信息。
- 将职位要求与用户简历进行结构化对比。
- 输出可解释的匹配结论，而不是只给一个分数。
- 给出可执行建议，例如“适合强调哪段经历”“沟通时可以问什么”“这个岗位的主要风险是什么”。

## 5. MVP 范围

### 5.1 简历上传

用户可以在插件设置页或侧边栏上传简历文件。

支持格式：

- MVP 必选：PDF、纯文本。
- 后续可选：DOCX。

上传后，插件将简历内容解析为结构化数据，包括：

- 基本信息：姓名、联系方式、城市。
- 教育经历。
- 工作经历。
- 项目经历。
- 技能关键词。
- 行业、岗位、年限等概括信息。

MVP 不需要支持多份简历，但数据结构应允许未来扩展。

### 5.2 求职意向输入

用户可以输入目标岗位，例如：

- 前端工程师
- AI 产品经理
- 数据分析师
- Java 后端开发

MVP 中只保留“目标岗位”。目标城市、目标薪资、期望行业、目标职级等字段放到后续版本迭代。

### 5.3 职位页面识别

当用户访问支持的网站职位详情页时，插件自动识别页面内容。

MVP 支持：

- Boss 直聘职位详情页。
- 用户正在浏览的职位详情自动捕捉，不要求用户手动粘贴 JD。

后续扩展：

- 猎聘。
- 拉勾。
- 智联招聘。
- 前程无忧。

页面提取字段：

- 职位名称。
- 公司名称。
- 薪资范围。
- 城市与办公地点。
- 经验要求。
- 学历要求。
- 职位描述。
- 技能关键词。
- 加分项。
- 公司行业或规模。

### 5.4 匹配分析

插件根据简历、求职意向和当前职位输出：

- 总体匹配等级：高 / 中 / 低。
- 匹配原因：3 到 5 条。
- 不匹配或风险点：1 到 5 条。
- 简历可突出内容：从用户简历中选出最相关经历。
- 沟通建议：用户和招聘方沟通时可以问的问题。
- 投递建议：建议投递 / 谨慎投递 / 不建议投递。

分数可以作为辅助展示，但不能作为唯一结论。MVP 优先输出可解释文本。

### 5.5 页面内呈现

MVP 推荐采用插件侧边栏或页面右侧浮层。

基本结构：

- 顶部：当前职位名称、公司、匹配等级。
- 中部：匹配点、风险点、简历亮点。
- 底部：沟通建议、重新分析按钮、设置入口。

交互原则：

- 不遮挡招聘网站原页面主要内容。
- 不自动点击、投递、发送消息。
- 用户可以手动关闭或暂停当前网站触发。

## 6. 非目标范围

MVP 不做以下能力：

- 自动投递简历。
- 自动发送聊天消息。
- 自动批量爬取职位。
- 绕过招聘网站限制或登录机制。
- 代替用户做最终求职决策。
- 长期保存用户敏感信息到未经说明的第三方服务。

这些功能可能带来账号风险、平台风控、隐私合规和用户信任问题，应在后续版本单独评估。

## 7. 技术可行性

### 7.1 Chrome 插件实现

插件可以用 Manifest V3 实现，主要模块包括：

- Content Script：读取招聘网站页面上的职位信息，并向页面注入侧边栏或浮层。
- Extension Service Worker：处理插件事件、消息转发和后台任务。
- Options Page：上传简历、填写求职意向、管理设置。
- Storage：保存简历解析结果、求职意向和用户设置。

实现难点不在插件框架本身，而在网页结构适配、职位信息抽取、简历解析质量、AI 分析稳定性和隐私设计。

### 7.2 简历解析

难度：中到高。

原因：

- PDF 简历版式差异大。
- 中英文、表格、双栏、图形化简历都可能影响文本提取。
- 简历解析需要结构化，不只是纯文本读取。

MVP 建议：

- PDF 和纯文本的原始文本提取不依赖模型，优先使用本地解析库和文本清洗算法完成。
- 使用规则算法抽取明确字段，例如邮箱、手机号、时间范围、技能关键词。
- 使用 AI 模型参与结构化整理，将简历文本归类为教育经历、工作经历、项目经历、技能和个人摘要。
- 允许模型输出 JSON，但必须做 schema 校验和必要的兜底修正。
- 允许用户查看和修正解析结果。

说明：

- 如果完全不用模型，也可以生成基础 JSON，但质量主要依赖规则，对非标准简历、项目描述和岗位语义总结的效果会偏弱。
- MVP 推荐采用“本地文本提取 + 规则预处理 + 模型结构化”的混合方案。

### 7.3 职位页面抽取

难度：中。

原因：

- 招聘网站 DOM 结构可能变化。
- 不同职位页信息展示不一致。
- 有些内容可能需要登录后才能看到。

MVP 建议：

- 先只适配 Boss 直聘的典型职位详情页。
- 抽取失败时允许用户手动复制职位描述作为兜底。
- 将不同网站的抽取逻辑隔离，避免后续扩展时互相影响。

### 7.4 匹配分析

难度：中。

原因：

- 只靠关键词匹配会很粗糙。
- 只靠大模型生成可能不稳定。
- 用户需要的是可解释建议，不是泛泛而谈。

MVP 建议：

- 先用规则抽取硬性条件，例如城市、学历、年限、技能。
- 再用 AI 生成匹配解释和建议。
- 输出必须引用简历和岗位中的具体依据。

### 7.5 隐私与合规

难度：高。

原因：

- 简历包含姓名、电话、邮箱、教育和工作经历，属于敏感个人信息。
- 如果使用云端 AI，需要明确告知数据会被发送到哪里、用于什么目的、保存多久。
- 如果要上架 Chrome Web Store，需要准备隐私政策和数据使用说明。

MVP 建议：

- 默认只保存解析后的必要字段。
- 明确提供“删除简历数据”入口。
- 在发送给 AI 前尽量减少无关个人信息。
- 在产品文档和设置页中说明数据流向。

## 8. 产品必要性判断

### 8.1 为什么值得做

- 求职场景高频且决策成本高。
- 用户已经在职位页面做判断，插件能嵌入现有工作流。
- 简历和职位匹配天然适合结构化分析与 AI 辅助。
- MVP 可以先从“单网站 + 单简历 + 页面内建议”做起，验证成本可控。

### 8.2 主要风险

- 平台页面结构变化导致抽取失效。
- 招聘网站可能对自动化行为敏感。
- 用户对简历隐私非常敏感。
- AI 输出如果太泛，会让用户觉得没有价值。
- 简历解析错误会影响后续分析可信度。

### 8.3 成功前提

- 第一版分析结果必须具体、可解释。
- 用户必须能控制自己的简历数据。
- 插件不能干扰招聘网站正常使用。
- 不碰自动投递和自动聊天，先避免高风险能力。

## 9. 版本规划

### V0.1 Boss 直聘自动捕捉原型

目标：验证真实浏览场景中的核心链路是否成立。

功能：

- 上传 PDF 简历。
- 提取简历文本。
- 输入目标岗位。
- 在 Boss 直聘职位详情页自动捕捉职位信息。
- 在页面右侧展示匹配分析。
- 抽取失败时提供手动粘贴职位描述的兜底入口。
- 输出匹配分析。

验收标准：

- 用户访问典型 Boss 直聘职位详情页时，插件能自动识别职位名称、公司和职位描述。
- 插件能基于当前职位、简历和目标岗位输出匹配分析。
- 页面侧边栏不遮挡原页面关键内容。
- 输出包含匹配点、风险点和沟通建议。

### V0.1 后端补充记录（2026-06-16）

目标：把简历 AI 结构化和职位匹配分析从插件端迁移到本地后端，先跑通本地链路。

功能：

- 本地 Node.js + Express 后端监听 `localhost:3000`。
- `POST /api/resume/structure` 接收简历原文并返回结构化 JSON。
- `POST /api/match/analyze` 接收简历、目标岗位和当前职位，并返回匹配分析 JSON。
- 模型 API Key、模型名和 Base URL 只从后端 `.env` 读取，插件端不保存模型 API Key。
- 设置页展示后端返回的结构化简历，并允许用户编辑后保存。
- Boss 职位抽取逻辑不在本阶段改动。

验收标准：

- 插件能把简历文本发送到本地后端并保存结构化结果。
- 插件能把当前职位、目标岗位和结构化简历发送到本地后端并展示匹配分析。
- 插件能保存最近 20 条分析历史，并在侧边栏展示最近 5 条。
- 结构化结果包含姓名、联系方式、所在地、教育、工作、项目、技能和摘要。
- 模型返回非严格 JSON 时后端自动重试一次。

### V0.1 当前状态补充记录（2026-06-16）

当前版本已经完成本地 MVP 主链路：简历上传、简历结构化、本地后端 AI 匹配、Boss 普通职位捕捉、侧边栏结果展示、分析历史和数据隐私控制均已打通。

已完成能力：

- 本地后端固定运行在 `localhost:3000`，API Key、模型名和 Base URL 只保存在 `.env`。
- 插件端保留 PDF/文本简历提取，简历 AI 结构化和职位 AI 匹配都迁移到后端。
- Boss 普通职位的职位名、薪资、经验、学历和 JD 捕捉整体可用。
- 侧边栏已改为紧凑工作台样式：当前职位摘要、AI 匹配结论、主要风险、分数拆解、详情折叠、最近分析历史。
- 匹配结果采用结论优先展示：投递建议、分数/等级、进度条、摘要、主要风险优先，长解释默认收起。
- 最近分析历史支持查看详情、收藏和收起详情。
- 设置页支持结构化简历编辑、删除简历数据、清空分析历史和数据使用说明。

仍未完成或需要后续验证：

- Boss 猎头职位的委托公司字段仍不稳定，需要真实样例回归和字段来源调试。
- 扫描版 PDF OCR 暂未支持。
- 尚未建立系统化 Prompt/模型输出回归集，AI 分析质量主要依赖人工体验验证。
- 侧边栏视觉已经可用，但信息密度和结果呈现仍适合继续小步迭代。
- 当前只面向本地个人使用，暂不考虑部署、账号体系或 Chrome Web Store 上架。

## 11. 当前产品决策

已确认：

- 当前阶段只做个人本地使用，不做部署和多用户服务。
- 允许调用 OpenAI-compatible 云端模型，但必须经由本地后端代理，插件端不保存 API Key。
- V0.1 以 Boss 直聘为唯一适配网站，普通职位作为稳定基线。
- 输出定位是“求职决策辅助”，分数只是辅助，重点是解释、风险和投递建议。
- 简历原文、结构化简历和分析历史保存在 `chrome.storage.local`，用户可以在设置页删除。
- 不做自动投递、自动聊天、批量爬取，也不绕过招聘网站登录或风控。

仍待后续决定：

- 是否需要支持多份简历和不同求职方向的组合管理。
- 是否需要 OCR 支持扫描版 PDF，以及 OCR 使用本地能力还是外部模型/API。
- 是否需要把分析历史升级为独立的职位管理列表。
- 是否需要扩展到猎聘、拉勾、智联等其他招聘网站。
- 如果未来给他人使用，隐私政策、数据保留策略和 Chrome Web Store 合规材料需要单独设计。

## 12. 后续迭代方向

### 12.1 短期：稳定当前本地 MVP

目标：让现有链路在真实使用中稳定、可信、好判断。

建议优先级：

1. 建立 Boss 真实样例回归集，普通职位和猎头职位分开记录。
2. 增加字段来源调试模式，定位每个字段来自结构化数据、标题区、左侧卡片、正文还是兜底。
3. 专项处理猎头职位委托公司；来源不确定时显示“未公开/待确认”，不要硬猜。
4. 建立 AI 输出回归样例，用固定简历和固定 JD 检查 JSON 合法性、编造问题、风险漏判和建议泛化。
5. 继续压缩侧边栏信息密度，优先优化“第一眼能否判断要不要投递”。

### 12.2 中期：提升求职决策价值

目标：从“能分析”走向“分析稳定且能指导行动”。

可选方向：

- 结果解释质量：让匹配点、风险点和简历亮点都绑定简历/JD 依据。
- 沟通辅助：基于风险点生成可复制的招聘方提问清单。
- 职位历史：支持收藏、筛选、备注和对比多个岗位。
- 简历适配建议：提示当前岗位应该突出哪段经历，但不直接生成虚构内容。
- 用户偏好：增加目标城市、薪资、行业、职级等筛选条件。

### 12.3 后期：能力扩展与合规

目标：在主链路验证后，再扩展复杂能力。

暂缓事项：

- 扫描版 PDF OCR。
- DOCX 简历解析。
- 多招聘网站适配。
- 多份简历管理。
- 云端部署、账号体系和同步。
- Chrome Web Store 上架。
- 自动投递、自动聊天和批量抓取。

### 12.4 下一次建议开始的任务

下一轮建议先做“Boss 真实样例回归 + 字段来源调试”。原因是：普通职位已经基本可用，继续直接改猎头字段容易影响稳定基线；先把样例和字段来源做清楚，后续修复才有可验证标准。
